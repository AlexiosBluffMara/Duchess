Investigación ML / ML Research

Pushing Gemma 4 to the Edge

A multi-stage compression pipeline — from training-time quantization through post-training compression to hardware-specific export — making a frontier 2.3B-parameter vision-language model fit on a phone at $0/inference.

The Duchess Quantization Pipeline
Gemma 4 E2B (BF16, 10.2 GB) → Unsloth Dynamic QLoRA (4.1 GB VRAM training) → LoRA Merge → AWQ Post-Training Quantization (mixed INT8/INT4) → LiteRT Export (Tensor G4 NPU, ~2.8 GB final)

Adapter merge MUST happen before post-training quantization. Quantizing first then merging produces double quantization error or defeats compression entirely.

⚡

1. Unsloth Dynamic QLoRA

QLoRA (Dettmers et al., NeurIPS 2023) quantizes base weights to 4-bit NormalFloat (NF4) — an information-theoretically optimal data type for normally-distributed neural network weights — while training full-precision LoRA adapter matrices:

W' = Wfrozen_NF4 + (α/r) · B·A    where A ∈ Rd×r, B ∈ Rr×d, r « d

Unsloth's "Dynamic" enhancement adds two innovations: (1) per-layer precision allocation — profiles each layer's Hessian trace and assigns bit-widths dynamically, keeping high-sensitivity layers at 8-bit while aggressively quantizing tolerant middle layers to NF4; (2) fused Triton kernels that merge NF4 dequantization + LoRA forward into a single GPU kernel, fuse RoPE computation into the attention kernel, and compute cross-entropy in-place without materializing the vocabulary-dimension logit tensor.

Memory comparison — Gemma 4 E2B fine-tuning (2.3B effective, 5.1B total with MoE)
ComponentFull BF16Standard QLoRAUnsloth Dynamic
Base weights10.2 GB2.65 GB~2.4 GB
LoRA adapters (r=16)—26 MB26 MB
Optimizer (Adam)40.8 GB104 MB104 MB
Activations (seq=2048, bs=4)3.2 GB3.2 GB~1.6 GB
Total VRAM54.2 GB6.0 GB4.1 GB

LoRA targets for VLMs: All 7 linear projections per transformer block (q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj). FFN layers store domain knowledge — including them improves construction safety accuracy by 2-4% over attention-only LoRA. The ViT vision encoder is not adapted: SigLIP features generalize well, and perturbing vision representations without retraining the cross-modal projection layer risks breaking alignment. For MoE: LoRA applied to ALL experts, not just the router.

Accuracy on multimodal tasks: The "0% loss" claim is validated on text benchmarks only. For VLMs, cross-modal attention is more sensitive to quantization noise (0.5-2% degradation on VQA, Shen et al. 2024). We mitigate by keeping the vision-language projection and first/last transformer layers at 8-bit precision.

📐

2. TurboQuant — Post-Training Quantization

Post-training quantization compresses a trained model without retraining. We benchmark three complementary techniques on the Tensor G4 NPU:

GPTQ (Frantar et al., ICLR 2023): Hessian-based weight quantization. For each layer, solves argmin ‖WX - WqX‖² using the inverse Hessian to compensate for quantization error — adjusting remaining weights in each row after quantizing each column. Processes a 7B model in ~4 minutes. Best at extreme quantization (INT3).

AWQ (Lin et al., MLSys 2024): Activation-Aware Weight Quantization. 1-3% of weight channels are "salient" (process high-magnitude activation channels). AWQ applies per-channel scaling to protect salient weights before uniform quantization. Faster than GPTQ (no Hessian), more hardware-agnostic, slightly better at INT4.

SqueezeLLM (Kim et al., ICML 2024): Dense-and-sparse decomposition. Isolates outlier weights (~0.5%, high Fisher information) into a sparse FP16 matrix, then aggressively quantizes the dense remainder to INT3/INT4. Recovers most accuracy lost from extreme compression.

Hardware precision support — target deployment devices
PrecisionTensor G4 NPU (Phone)Snapdragon XR1 (Glasses)

Thermal envelope: Pixel 9 Fold sustains INT8 NPU inference at ~3.5W indefinitely (TDP ~5W, no throttling). FP16 on GPU draws ~5.5W and throttles after ~90s to ~60% perf. Vuzix M400 (600mAh, 2.5W TDP) must minimize duty cycle — inference only on flagged frames.

Our recommendation: INT8 weights + FP16 activations for attention layers, INT4 weights + INT8 activations for FFN/expert layers. Combined with PrismQuant layer allocation, this yields ~2.8 GB on-device with minimal accuracy loss.

🔬

3. BitNet b1.58 — 1-Bit Ternary Quantization

Research frontier. Ma et al. (Microsoft Research, 2024) constrain every linear layer weight to {-1, 0, +1} via absmean quantization. Each weight carries log2(3) = 1.58 bits of information. Packed storage: 5 ternary values in 8 bits (35 = 243 < 256).

Why this matters: Ternary multiplication becomes pure addition/subtraction — no FPU required:

yj = Σ{W=+1}(xi) − Σ{W=−1}(xi)

At 7nm: ternary add/sub costs ~0.005 pJ vs FP16 multiply at ~0.15 pJ — a 30× energy reduction (Horowitz, ISSCC 2014). On ARM NEON (Snapdragon XR1), each 128-bit instruction processes 16 INT8 values via the integer SIMD pipeline, bypassing the FPU entirely. Published results on 3.9B models: 2.71× faster, 3.55× less memory, 71.4× more energy efficient for matmul operations.

Memory for Gemma 4 E2B: Linear layers at 1.6 bits/weight = 840 MB; embeddings at FP16 = 1.8 GB. Total: ~2.64 GB (vs 10.2 GB BF16, vs 2.87 GB NF4). The real win is computational, not storage.

The fundamental challenge: BitNet achieves these results by training from scratch with ternary constraints. Post-training ternarization causes catastrophic accuracy loss (MMLU drops ~46% → ~26% on Llama 2 7B). We cannot simply ternarize our fine-tuned Gemma 4.

Vision pathway concern: No published results exist for ternary VLMs. The ViT patch embedding encodes subtle visual features (edge orientations, texture gradients) through small weight differences that ternary collapses. Our proposed hybrid: ViT encoder at INT8, vision-language projection at FP16, language decoder FFN at BitNet b1.58.

Planned experiments: (1) Hybrid ternarization of decoder FFN layers only (~60% of params); (2) Ternary LoRA adapters (novel — B and A matrices with ternary constraints); (3) Knowledge distillation into a 1B BitNet student from Gemma 4 E4B teacher; (4) Custom ARM NEON ternary matmul kernels for Snapdragon XR1.

🔮

4. PrismQuant — Mixed Precision for MoE

Layer sensitivity (approximated by ‖W‖F · ‖H−1‖F) varies dramatically across transformer components. For Gemma 4's Mixture-of-Experts architecture, this means:

ComponentSensitivityPrecisionRationale
Embeddings / LM headVery HighFP16Direct impact on token probabilities
MoE Router (gating)CriticalFP16Quantization errors flip discrete routing decisions
Attention Q/K projectionsHighINT8KV-cache errors accumulate over sequence length
Attention V/O projectionsModerateINT8Less sensitive than Q/K
FFN / Expert layersLowINT4Over-parameterized, redundant; ~67% of total params
First/last 2 decoder layersHighINT8Adjacent to embeddings/logits; errors propagate

MoE-specific insight: Expert utilization is uneven — some experts activate far more frequently. High-frequency experts handle common tokens and are robust to quantization. Low-frequency experts handle rare/specialized tokens, are less regularized, and counterintuitively tolerate more aggressive quantization because the model doesn't rely heavily on their precise values.

VLM Fine-Tuning Format

Detection datasets provide bounding boxes, not natural language. For Gemma 4 vision fine-tuning, we convert annotations to instruction-tuning pairs using the native function-calling format — teaching the model structured output directly, eliminating brittle JSON parsing at inference time:

{
  "image": "scene_0042.jpg",
  "instruction": "Analyze this construction scene for PPE violations.",
  "output": {
    "tool_calls": [{
      "name": "create_safety_alert",
      "arguments": {
        "violation": "no_hardhat", "severity": 3,
        "description_en": "Worker on scaffolding without hard hat — OSHA 1926.100",
        "description_es": "Trabajador en andamio sin casco — OSHA 1926.100"
      }
    }]
  }
}

Training data mix: 40% EN safety scenarios, 30% ES construction-register scenarios (including code-switching), 20% visual reasoning chains, 10% general instruction following (prevents catastrophic forgetting). All bilingual data uses construction-register Spanish — casco duro not casco de seguridad, montacargas not carretilla elevadora.

A Universal Architecture for Situated Intelligence

Glasses see. Phone thinks. Cloud scales. This three-tier pattern — built on Gemma 4 — generalizes to any domain requiring real-time, on-device visual AI.

The Core Pattern: See → Think → Act

👓
Tier 1 — Glasses
Camera + display. Edge detection. Scene capture.
📱
Tier 2 — Phone
On-device VLM inference. Real-time reasoning. $0/query.
☁️
Tier 3 — Cloud
Batch analytics. Escalation. Dashboards.

Military & Defense — Vuzix M400

The Vuzix M400 is already deployed with US military and NATO allies. Ruggedized (IP67, MIL-STD-810G), it supports thermal/IR camera modules. Vuzix holds contracts with the US Department of Defense, and the M400 is used by the US Army, Air Force, and defense contractors including BAE Systems, L3Harris, and Elbit Systems.
